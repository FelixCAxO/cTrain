# cTrain Architecture

cTrain (Code Training) is a VS Code extension for typing lessons, existing files, and selections in editable `code-trainer:` buffers.

The extension registers `cTrain: Start Lesson`, `cTrain: Practice Current File`, `cTrain: Mock Exam`, `cTrain: Pause Session`, `cTrain: Restart Session`, and `cTrain: Session Actions`, shows a persistent status bar entry on activation, and contributes a welcome view plus editor context menu entry for practicing the current selection or file. It opens a writable virtual document through the `code-trainer` scheme. A writable virtual file system provider is used instead of a read-only text document content provider so the learner can type directly in the lesson buffer. Authored lesson JSON supports C, Java, and Python, while current-file practice preserves the source editor language id. The editor shows the document returned by VS Code after that language assignment so the user's installed local theme and language grammar provide normal syntax colors for typed text.

Built-in lesson assets are loaded recursively from `lessons/`, with authoring-only folders such as `lessons/_schema/` ignored by the loader. Catalogue ordering is shared by the built-in lesson list, lesson picker, lesson tree, and completion `Next Lesson` action through the same comparator: Java Foundations, Java 25 Cert Exam, Java 26, C, Python, Prog2 references if present, then future fallback tracks; within a track, direct prerequisite edges sort before numeric id suffixes and id tie-breaks. Completion actions use `LessonCatalogue` to expose adjacent lesson flow from that order. Load-time validation rejects duplicate ids, missing prerequisite ids, cyclic prerequisite chains, and lessons that depend on a rejected prerequisite so locked lessons cannot silently become unreachable. Lesson metadata uses documented C, Java, and Python `languageVersion` labels so authoring conventions stay stable. The public catalogue includes Java certification categories, C systems-foundations lessons, and Python PCEP/PCAP/PCPP1 categories. Completion checks are used after lessons and as the Java certification-focused question bank for `cTrain: Mock Exam`; displayed choices are shuffled at runtime while preserving the authored answer mapping, and non-Java lessons plus off-syllabus preview, Java 26, and HTTP client side-track lessons are excluded from mock exams by default. Mock exams follow the current Java SE 25 format: 50 questions, 120 minutes, a 68% pass line, and objective-weighted sampling across the certification blueprint. The repository keeps C, Java, and Python public lessons plus 36 Prog2 reference lessons; the reference lessons live in `prog2-lessons/`, reuse the same runtime loader and comparator, and are excluded from the published VSIX.

`cTrain: Practice Current File` uses the active editor. If text is selected, the selection becomes the training target; otherwise the whole file becomes the target. Sources under 20 characters are rejected, and sources over 20,000 characters require confirmation before a lesson opens. The original file is never cleared or overwritten. Source-practice lessons add a per-open session suffix to their virtual lesson id so repeated current-file runs do not reuse a dirty `code-trainer:` document from an earlier source. The typed buffer is checked by `TrainingSession`, which delegates prefix correctness to `prefixMatch.ts` and paste policy to `pastePolicy.ts`. After a valid prefix reaches the end of a target line, the session automatically consumes the target newline characters and syncs the virtual document, including consecutive newlines for blank rows, so learners can continue with the next code character. Document sync moves the cursor by logical lesson position rather than raw string offset, so CRLF-backed VS Code documents still land on the next target row. Opening and restarting a session also synchronize the visible virtual document to the session's initial text and place the cursor at the end of that initial lesson text, which matters for generated leading comment headers before the first typeable code line. If VS Code leaves the next typed code character inside the auto-consumed newline run, the session recovers it to the current lesson cursor before prefix matching. Stale replacement events that would move progress back to an older prefix are rejected unless they are explicit deletion or undo/redo edits. Closing braces typed a few spaces before their indentation target are recovered by filling only the missing leading whitespace, and stale auto-closed braces are ignored only while pending automatic indentation proves where the real typed brace belongs.

Ghost text is decoration-only. The extension does not register an inline completion provider. One decoration is produced per remaining target line, with `after.contentText` carrying the visible prompt, the color coming from the active theme's `editor.foreground` token, and the configured opacity defaulting to `0.4`. Decoration prompt text is not language-tokenized by VS Code, but it now follows installed themes such as Catppuccin instead of a separate ghost-text palette. Future target-line segments stay in the decoration model instead of being dropped, which prevents newline-by-newline ghost jitter at lesson start. Folded ghost text is capped to three visible lines with a remaining-line marker so long source files do not drown the active line. Auto-inserted line breaks create the real document rows before the next ghost segment is rendered, so blank rows in the source remain visible while typing.

Paste is rejected by default when a single edit inserts more than one logical character and is not undo, redo, editor auto-indent, auto-close insertion, or a target-prefix insertion. The first rejection explains that paste is disabled to build muscle memory; later rejections use a shorter status message. Target-prefix insertions allow IntelliSense accept, snippet expansion, IME commits, keyboard-layout generated symbols, and on-type formatting rewrites to flow through normal prefix analysis when the resulting document is still a valid lesson prefix. Grapheme segmentation is used instead of UTF-16 length so composed single characters such as emoji or combining-accent input are not treated as paste. The paste classifier normalizes CRLF edit text and accepts a pending line-layout prefix before the next target character so layouts such as Swedish AltGr can type `}` after an auto-consumed line break without being mistaken for paste. Auto-close pair behavior is defined once in `src/training/autoClosedPairs.ts` and shared by paste classification and prefix matching so the two paths cannot drift silently. Users can enable `cTrain.allowPaste`, but training metrics assume the default.

Document-change handling is serialized per training document URI. Each VS Code text change snapshots the document text and edit metadata before entering the queue, so rapid typing cannot interleave document rollback, progress recording, and completion actions for the same lesson buffer. Decoration repaint is skipped when the live visible editor text has already advanced beyond the session snapshot, preventing stale ghost text from being painted over a newer queued keystroke. Programmatic synchronization edits are still ignored through the `revertingDocuments` guard. If VS Code rejects a text-editor synchronization edit, cTrain retries through a workspace edit before surfacing the error and clearing the affected training session.

Progress is stored in VS Code global state under `codeTrainer.progress.v2` with migration support for older blobs. Completion records include lesson id, lesson version, duration, completion time, typed characters, WPM, mistake count, rejected paste count, and detailed mistake characters. Per-lesson progress keeps attempt history, `lastWpm`, `bestWpm`, and a personal-best record with error rate. Wrong completion-check answers create review items for day 1, day 3, and day 7; due reviews are surfaced after lesson completion and tracked separately from fresh lesson answers so re-test accuracy can be measured. Mock exam attempts are stored separately with score, duration, lesson ids, objective-level results, and missed-question snapshots so exam history does not alter lesson streaks. Readiness is derived from the rolling last-5 mock average, the 3 consecutive 80% gate, and final-week objectives below 70%. Completion and exam writes are serialized through the progress store so overlapping writes do not lose records, and loaded progress blobs are cloned before mutation so a failed global-state update does not leak partial changes into memory. The progress blob also stores `lastCompletionDate`, `currentStreak`, and `longestStreak`; activation can show a streak reminder when yesterday had a completion and today has not.

The lesson picker and lesson tree group the built-in lessons into ordered curriculum sections: Java Foundations (ids `01`-`45`), Java 25 Cert Exam (`50`-`69` plus growth ids `80`+), Java 26 (`70`-`79`), C Foundations, and Python PCEP/PCAP/PCPP1. Catalogue order follows the same category bands, so `Next Lesson` finishes the whole Java cert band before reaching the Java 26 demos, then the C and Python tracks. The tree is the expandable surface; the picker is flat but uses matching separators and searchable category details on lesson rows. No lesson is gated. Lessons whose prerequisites are not yet complete show a non-blocking `Suggested after:` hint listing those prerequisites in suggested order, for example `Methods -> Interfaces -> Classes`, with independent paths comma-separated, but any lesson can start at any time. Completed lessons show a `×N` completion-count badge next to the name plus a personal-best WPM and error-rate summary. Rows also show due recall-review counts, and the picker title shows progress toward the 12-day, 900-minute Java 25 study sprint. Completion-time `Next Lesson` follows catalogue adjacency so a learner can continue forward in catalogue order, and Prog2 reference source-file lessons keep their prerequisite chain as pure ordering metadata with no hint, so learners can jump directly to the reference they need.

The status bar reports percent complete, WPM, elapsed time, and mistake count during a session. Paused sessions use a warning-colored `[PAUSED]` prefix so ignored typing has visible context. Restart creates a fresh `TrainingSession` for the same lesson, reseeds the virtual document, clears pause/decorations cache state, and resets document text, counters, mistakes, ghost text, and completion state. Mistakes temporarily add the expected character, keyboard hint for symbols, actual character, and line number; the in-editor mistake hover uses the same keyboard hint formatting. Completion shows a toast with elapsed time, WPM, and mistakes, plus `Next Lesson` and `Retry` actions.
